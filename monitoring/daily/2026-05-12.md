## P2-05 Mode A handling (judgment recorded)

`AI_WORKFLOW.md` §4 reserves Mode A (adversarial review by the Backup AI) for three triggers: (1) phase gates, (2) high-risk architecture decisions (new control surface, new trust boundary, non-reversible decision), and (3) safety-critical decisions (live trading authorization, broker integration, kill-switch behavior, operator heartbeat policy, data retention/destruction policy). Whether PR #18 (the P2-05 governance-only authorization slice) and PR #19 (the P2-05 implementation PR) trigger Mode A is a judgment call. This packet records that judgment explicitly so the audit trail is not silent on the question.

**Decision.** This monitoring PR records the explicit judgment that **no retrospective Mode A adversarial review is required for PR #18 or PR #19**, based on the following observations from the merged code and the authorization:

1. **PR #18 is governance-only**, exactly mirroring PR #14 (P2-03 authorization) and PR #16 (P2-04 authorization). It lands a durable authorization artifact and extends the §8 step 4a allowlist by exactly one path (`src/gmc_rebuild/risk`); it does not materialise the directory and lands no code. Mode A is explicitly **not** triggered by routine governance documentation edits per `AI_WORKFLOW.md` §4 ("Do not invoke the backup AI in Mode A for routine documentation edits, template tweaks, dependency bumps that match committed versions, or typo fixes").
2. **PR #19 is types/interfaces-only.** The diff (`src/gmc_rebuild/risk/__init__.py`, `src/gmc_rebuild/risk/interfaces.py`, `tests/test_package_skeleton.py`, `tests/test_risk_interfaces.py`) contains exactly: three `Enum` classes, three frozen `slots=True` dataclasses, three `@runtime_checkable` `typing.Protocol` definitions, one `RiskControlError` exception, one `to_utc_string` helper, and tests. There is no concrete implementation of any protocol inside the runtime package, no broker SDK import, no network call, no filesystem call, no `__main__` entry point, no background thread, no scheduler, no daemon, no order object, no order placement, no market-data ingestion, no env-var loading, no secrets, and no real account / venue / broker identifiers.
3. **No new control surface, no new trust boundary, no non-reversible decision.** The kill-switch / reconciliation / heartbeat protocols defined in `interfaces.py` describe the *shape* of future runtime boundaries; they do not establish runtime behavior. ADR-002, ADR-003, and ADR-005 already define the policies; PR #19 only declares Python `Protocol` types matching those policies. Reverting PR #19 is `git revert <merge>` and is fully reversible because no external system has been touched.
4. **Not safety-critical in `AI_WORKFLOW.md` §4 sense.** The §4 examples of safety-critical decisions are "live trading authorization, broker integration, kill switch behavior, operator heartbeat policy, data retention or destruction policy". PR #19 defines no behavior for any of those; it defines the abstract Python types that a future, separately-authorized PR would have to implement. The behavior decisions live in ADR-002 / ADR-003 / ADR-005, which were each already addressed by their own ADR review (ADR-008 covers Mode A coverage for those ADRs prospectively, not retrospectively).
5. **The P2-05 authorization explicitly forbids the Mode A triggers.** `governance/authorizations/2026-05-12_p2-05.md` §"Explicitly Not Authorized" enumerates broker SDK integration, order placement, live or paper trading, market-data ingestion, persistent storage, schedulers / daemons, env-var loading, real secrets, and concrete implementation of the protocols inside the runtime package. A PR that observes all those constraints — as PR #19 does — does not by itself reach the §4 thresholds.

This judgment is recorded here in the Mode B packet only. It is **not** a Mode A critique and does **not** substitute for one; it is a documented decision that the §4 triggers are not met. Per `AI_WORKFLOW.md` §4 "Overlap" and ADR-008 §D7, if Kevin or Perplexity Computer wants an explicit Mode A critique on PR #18 or PR #19 retrospectively, it would be delivered as PR-review text (not committed to the repo) and would not require re-opening the merged PRs. The default of leaving the Mode A box unchecked on PR #18 and PR #19 stands unless Kevin reverses this judgment, in which case the follow-up below is the path. This judgment is **not** a green light to skip Mode A on a future PR that opens Phase 3, wires a concrete broker integration, materialises a runtime kill-switch implementation, or adds an operator-heartbeat daemon — each of those would meet §4 on its own and must be reviewed in Mode A before merge.

If Kevin disagrees with this judgment, the follow-up is a Mode A critique delivered as PR-review text on either PR #18 or PR #19 (or on a successor PR that touches the same surface), per `AI_WORKFLOW.md` §4 "Overlap" and §5; no in-tree artifact is required for the critique itself, but its existence and Kevin's decision following it should be recorded in the next monitoring packet.
