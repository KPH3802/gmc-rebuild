They are recorded here for carry-forward so they are captured in the repository (per the `RECOVERY.md` §15 "No Chat-Only Risk Rule" introduced by GOV-01) and so the maintainer can address them in a future docs/governance cleanup packet **before** the first execution-adjacent packet opens. **They do not block P5-02 merge and they do not require any change to PR #107.**

**F1 — Future price-representation control.** `SimulatedOrderIntent.limit_price` is currently `float | None`. Binary floating-point is the classic financial-software anti-pattern for prices (representational error at common decimal values). For an inert local-only data placeholder it is acceptable, and the verbatim authorization explicitly permits `float` ("optional limit price if appropriate"). The strict per-field validation rejects `bool`, `int`, `nan`, `±inf`, zero, and negative values for `LIMIT`. **Carry-forward control:** any future execution-adjacent path (paper broker, live broker, audit trail intended for reconciliation, tax artifact, etc.) must revisit the price-representation decision and switch to `Decimal` or a fixed-point representation under its own separate written authorization. A future docs/governance cleanup packet should extend `RECOVERY.md` §16.5 (or §16.6) with an explicit `Future price-representation control` entry naming this requirement so it is not lost when the first execution packet is opened.

**F2 — Future symbol validation control.** `SimulatedOrderIntent.symbol` is currently a non-empty whitespace-free `str`, opaque to this module (no symbol-universe lookup, no exchange resolution, no normalization to any vendor format, no validation against any market-data source). For an inert opaque placeholder this is acceptable, and the verbatim authorization explicitly permits the opaque shape. **Carry-forward control:** any future packet that wires `symbol` to anything real (broker, market-data feed, persistence, audit trail) must add strict validation, normalization, length cap, character allowlist, and vendor mapping under its own separate written authorization. `RECOVERY.md` §16.5 already records the symbol-universe gap implicitly; a future docs/governance cleanup packet should make the symbol-validation requirement explicit at the same time as F1.

**F3 — Future quantity-bound control.** `SimulatedOrderIntent.quantity` is currently a positive `int` (with `bool` and `float` explicitly rejected). It is unbounded above — a `SimulatedOrderIntent(quantity=10**18)` is constructable. For inert local-only data this is harmless, and the verbatim authorization explicitly permits unbounded positive integers. **Carry-forward control:** any future execution-adjacent path must define a max-quantity guard, a per-symbol cap, a notional-value cap (which itself implies a price-resolution control — see F1), or a similar bound under its own separate written authorization. A future docs/governance cleanup packet should extend `RECOVERY.md` §16.5 with an explicit `Future quantity-bound control` entry so the requirement is captured.

None of F1 / F2 / F3 introduce any forbidden behaviour, weaken the P4 safety foundation, open any unauthorized P5 work, change the inert character of the `src/gmc_rebuild/simulation/` package, or change the contract of `SimulationBoundary.propose_order`. They are forward-looking technical notes about the eventual move from inert simulation to execution, recorded in this monitoring packet so the next governance cleanup packet has a captured list of items to fold into `RECOVERY.md` §16.5 (or §16.6) under its own separate written authorization.
