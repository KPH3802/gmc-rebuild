Date: 2026-05-16
Authorizer: Kevin Heaney
Scope: Bounded, docs-only operations packet for `gmc-rebuild` documenting (a) an explicit OPS roadmap covering OPS-01 through OPS-07 with each item's current status, scope, and authorization basis, and (b) a read-only Mac Studio memory-pressure baseline procedure with pass / fail / escalation criteria and an explicit P5 gate. Makes no other changes.

  - **§14.1 Why this baseline exists** — restates the OPS-03 finding that the 2026-05-13 ungraceful reboot had a stronger evidence pattern for memory pressure plus a UI hang than for any X10 Pro storage fault.
  - **§14.2 Read-only baseline procedure** — defines a before-heavy-work snapshot, a during-heavy-work snapshot, an optional after-heavy-work snapshot, the four tools used (`vm_stat`, `memory_pressure`, `top -l 1 -o mem -n 30`, Activity Monitor), and the storage rule (operator-side, outside this repository).   - **§14.3 Escalation criteria** — defines five escalation signals (recurring `JetsamEvent` artifacts, sustained swap pressure, sustained compressed-memory pressure, repeated non-storage UI hangs, another ungraceful reboot) and states that escalation only triggers across more than one observation; a triggered escalation requires a separate, written, operator-authorized OPS-03-shape investigation packet, which is **not** authorized by this packet.
  - **§14.4 P5 gate** — states that Phase 5 (local/paper simulation, when authorized) may proceed only after OPS-04 is either (i) completed per §14.2 with the result recorded as a dated entry under §5.6 under its own separate future written authorization, or (ii) explicitly waived by Kevin in a separate written authorization that names this §14.4 gate. The gate is additive to every other §10 unresolved risk and every other authorization required by `AI_WORKFLOW.md` §7; satisfying §14.4 alone does not authorize P5.
