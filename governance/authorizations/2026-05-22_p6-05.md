## Authorized Implementation Shape (For the Future P6-05 Implementation PR)

The future P6-05 implementation PR — opened separately, from clean `main`, on its own feature branch, under this authorization — is authorized to make only the following repository changes, and only within the contract below. Each numbered choice resolves an open decision the planning packet left to this authorization.

### 1. State Model — Frozen / Value-Typed Replaceable Snapshot (Authorized)

The simulated portfolio state is authorized as a **frozen, slotted, value-typed replaceable snapshot** (planning packet §3.3 candidate model (1)). It is **not** an event-sourced or append-only event model (planning packet §3.3 candidate model (2) is **not** authorized).

- The portfolio snapshot is a frozen, slotted dataclass holding a closed mapping of `symbol -> position` (where a position is itself a frozen, slotted value type with the closed fields the implementation names — see §6) plus a duplicate-prevention structure (see §2).
- "Applying" an accepted intent is a **pure transformation that returns a new snapshot**; the prior snapshot is never mutated in place. State evolution is expressed by replacing one immutable snapshot with the next.
- This mirrors the frozen/slotted discipline already established by the merged P5-02 / P6-04 `SimulatedOrderIntent`, P6-01 `SignalIntent`, P6-02 `EligibilityDecision`, and P6-03 `PositionDecision`.

### 2. Idempotence Model — Idempotent Application by Deterministic Simulated Order Intent ID (Authorized)

Application is authorized as **idempotent, keyed on the deterministic P6-04 simulated order intent ID**.

- The snapshot carries a **deterministic applied-intent-ID set** (or an equivalent value-typed duplicate-prevention structure — e.g. a sorted tuple of applied intent IDs). The exact representation is an implementation detail, provided it is value-typed, deterministic, and frozen/slotted-compatible.
- Re-applying the **same** accepted P6-04 `SimulatedOrderIntent` (identified by its deterministic intent ID, e.g. from `derive_simulated_order_intent_id`) to a snapshot that has already applied that ID **must not double-apply** the position change: the second application returns a snapshot equal to the first application's result (the duplicate is recognized and absorbed, not double-counted).
- Applying a **distinct** accepted intent (distinct intent ID) applies its position change normally and records its ID.
- This is the §3.3.1 candidate "idempotent / dedup-on-identity" contract; the explicitly-non-idempotent event-style contract is **not** authorized.

### 3. Authorized Directory Name and §8 Forbidden-Token Reconciliation

The authorized directory name is **`src/gmc_rebuild/portfolio_state/`** (with tests under **`tests/portfolio_state/`**). The candidate `src/gmc_rebuild/portfolio/` from the planning packet is **not** authorized, and `portfolio_state` is preferred because it makes the local-only simulated-state meaning explicit.

**Empirically verified at authoring time:** `portfolio_state` **still collides** with the `portfolio` forbidden-category token. The `MASTER_STATUS.md` §8 step 4 / step 4c forbidden-token scans tokenize each path component on `.`, `-`, and `_`, so `portfolio_state` tokenizes to `portfolio` + `state`, and `portfolio` is in the forbidden set. Running the exact §8 awk tokenizer at authoring time flags `src/gmc_rebuild/portfolio_state`, `src/gmc_rebuild/portfolio_state/__init__.py`, and `tests/portfolio_state` on the `portfolio` token (whereas a name with no forbidden component, e.g. `position_book`, scans clean). Choosing `portfolio_state` is therefore a deliberate decision to accept the same situation that already exists on `main` for `src/gmc_rebuild/signal_intake` (which tokenizes to `signal` + `intake`, and `signal` is forbidden), not an avoidance of it.

This is **not** a blocker. It is resolved exactly as the merged P6-01 `signal_intake` precedent resolved it. The future P6-05 implementation PR **must**, in the same PR that introduces the directory:

### 5. Closed Inputs (Authorized)

The future P6-05 surface accepts only local, deterministic, value-typed inputs already produced by prior Phase 6 layers, plus explicit caller-supplied fixtures. Every input is supplied by the caller as a value; no raw configuration slice, no environment variable, no filesystem, no broker session, no market-data snapshot, no real account, no time source.

- The **P6-03 accepted decision output / acceptance-verdict surface** — a `PositionDecision` from `gmc_rebuild.decision` (closed five-field shape: `outcome`, `reasons`, `intent_id`, `eligibility`, `verdict`). Only an **accepted** decision (`WOULD_TRADE`) results in a position change; a `WOULD_SKIP` / ineligible / blocked decision is consumed as a no-op (or recorded skip) and does not perturb positions.
- The **P6-04 `SimulatedOrderIntent`** from `gmc_rebuild.simulation` (closed nine-field shape including `time_in_force`), consumed unchanged.
- The **deterministic simulated order intent ID** (e.g. produced by the merged `derive_simulated_order_intent_id` helper or carried as the `intent_id` field), used as the idempotence / deduplication key.
- **Explicit fixture-only fill information**, if needed, supplied as a value-typed test/local input (e.g. a deterministic fill price and fill quantity, or a closed fill-rule value such as full-fill-at-limit). This is fixture-only and carries **no** market-data, broker, or routing capability.
- The **prior portfolio snapshot** supplied by the caller as a value (the first application may take an explicit empty / zero-position starting snapshot from a pure constructor).
- Optionally, a P6-01 `SignalIntent` and / or P6-02 `EligibilityDecision` carried forward by value for trace correlation only (no expansion of those merged shapes).

### 6. Closed Outputs (Authorized)

The future P6-05 surface produces only an **in-memory, value-typed simulated portfolio state object**:

- **Positions keyed by symbol** — a closed mapping `symbol -> position`, where a position is a frozen, slotted value type with the closed fields the implementation names (e.g. net quantity, and — only if the implementation design explicitly requires it — an average/applied price kept local and value-typed).
- **Deterministic quantity changes** computed from the accepted intent and the fixture-only fill information.
- A **deterministic applied-intent-ID set** (or equivalent value-typed duplicate-prevention structure) supporting the §2 idempotence contract.
- **No cash ledger** unless the implementation design explicitly requires it and keeps it strictly local and value-typed; if introduced, it must carry no real balance, no funding, no settlement, and no external state.
- **No P&L, no broker reconciliation, no balances, no live account, no market valuation, and no external state** of any kind.

Hashability is structurally inherited from the carried-forward values (mirroring the merged P6-03 `PositionDecision` precedent where unhashability of a carried `Mapping` field is inherited rather than papered over); the tests assert the inherited behavior, or the implementation may choose an explicitly hashable closed representation (e.g. a sorted tuple of position records) if it names that choice.

### 7. Required Implementation Constraints (Authorized Contract)

The future P6-05 implementation must be: **local-only; deterministic; pure / value-typed; in-memory; side-effect free**. It must have **no** file writes, **no** env-var reads, **no** network, **no** broker, **no** account identifiers, **no** real portfolio, **no** market data, **no** scheduler, **no** daemon, **no** runtime activation, **no** `__main__`, **no** secrets, and **no** tag/release.

- **No mutation of inputs.** The supplied prior snapshot, `PositionDecision`, `SimulatedOrderIntent`, fill fixture, and any carried-forward values are not modified in place. The new portfolio value type and any nested position value type are frozen / slotted.
- **No hidden state and no side effects.** No module-level mutable, no class-level mutable, no cache, no counter, no log buffer. No log emission inside the application function itself (structured audit logging is a P6-06-shaped concern, out of scope here). No `time.sleep`, no `asyncio.sleep`.
- **Closed import surface.** The internal module imports only `__future__`, stdlib `dataclasses` / `enum` / `typing` / `collections.abc` as needed, and the already-merged in-repo modules required to consume the typed inputs (`gmc_rebuild.decision`, `gmc_rebuild.simulation`, and optionally `gmc_rebuild.signal_intake` / `gmc_rebuild.eligibility` / `gmc_rebuild.runtime`). No import of `gmc_rebuild.config`, `os`, `socket`, `requests`, `urllib`, `http`, `threading`, `asyncio`, `subprocess`, `sqlite3`, `time`, `pickle`, `shelve`, `ssl`, `smtplib`, `ftplib`, `uuid`, `random`, or any third-party dependency.
- **No re-export from `src/gmc_rebuild/__init__.py`.** The new surface is reachable only via `from gmc_rebuild.portfolio_state import ...`, mirroring the merged P6-01 / P6-02 / P6-03 root-package non-re-export precedent.
