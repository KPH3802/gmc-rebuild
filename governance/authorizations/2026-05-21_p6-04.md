## Authorization

I authorize P6-04 implementation using Direction A.

The implementation should extend the existing local-only simulation order-intent surface in place. The goal is to make `SimulatedOrderIntent` more complete for dry-run use by adding deterministic order-intent identity and explicit time-in-force semantics.

This work should stay inside the existing simulation boundary. It should not create a new sibling package or a new dry-run subsystem.

## Plain-English Purpose

The system can already accept a signal, check eligibility, and compose a position/risk decision. The next concrete step is to make the simulated order object identifiable and more realistic for dry-run testing.

This does not mean live trading. It means the local test system can describe a fake order it would have considered, with a stable ID and time-in-force value, while remaining deterministic and fixture-only.

## Approved Scope

The builder may:

- Extend the existing `SimulatedOrderIntent` model in the simulation module.
- Add a deterministic intent-ID helper or equivalent deterministic identity mechanism.
- Add a closed `SimulatedOrderTimeInForce` enum.
- Update tests for the simulation boundary and order intent behavior.
- Update directly related documentation only if needed to keep status and module descriptions accurate.

## Expected Shape

The implementation should prefer:

- Keeping the change within the existing `src/gmc_rebuild/simulation/` module.
- Avoiding any new top-level `src/gmc_rebuild/<new_package>/` directory.
- Preserving immutability and deterministic behavior.
- Preserving local-only semantics.
- Maintaining compatibility with the existing P6-03 `PositionDecision` layer without coupling simulation to execution or broker concepts.

## Explicitly Not Authorized

This authorization does not allow:

- Broker integration.
- Live trading.
- Paper trading connected to a real broker.
- Market-data ingestion.
- Order placement.
- Account identifiers.
- Venue identifiers.
- Broker order IDs.
- Secrets or environment-variable reads.
- Network calls.
- Persistence or database writes.
- Runtime daemons.
- Schedulers.
- `__main__` entry points.
- Strategy logic.
- Portfolio accounting beyond what is already authorized elsewhere.
- A new sibling package for decided/composed order intents.
