## Proposed First Single-Protocol Test-Fixture Scope (Planning Level Only)

This section identifies the **proposed first single-protocol test-fixture scope** for the future P3-03 implementation PR at the planning level only. Naming a proposed scope here does **not** authorize implementation, does **not** name the specific protocol (Kevin names the specific protocol in writing at the time of P3-03 implementation authorization), and does **not** preclude Kevin from authorizing a different first single-protocol scope at the time of authorization. The proposal restates and refines `plan/phase3_entry_plan.md` §4 item 3 and the corresponding "Intended First Phase 3 Implementation Task Scope (Planning Level Only)" section of `governance/authorizations/2026-05-13_p3-02.md`, adding planning-level specificity that the Phase 3 entry decision deliberately deferred to this planning artifact.

**Proposed protocol selection (planning-level recommendation, not authorization).** The proposed first single-protocol fixture targets **exactly one** of the three P2-05 protocols:

- `KillSwitchProtocol` (ADR-002 — runtime kill switch).
- `ReconciliationProtocol` (ADR-003 — broker reconciliation discipline).
- `HeartbeatProtocol` (ADR-005 — operator availability heartbeat).

At the planning level only, `HeartbeatProtocol` is the **smallest safe first candidate** because:

1. Its abstract surface in `src/gmc_rebuild/risk/` is the smallest of the three (a single emit-style boundary plus supporting `HeartbeatStatus`/`HeartbeatRecord`-style dataclasses; no broker reconciliation state machine, no kill-switch trigger/clear semantics, no order-touching side effects to reason about).
2. A test-fixture exercise of `HeartbeatProtocol` can be authored as a pure-Python in-memory fake that emits a record in the test sandbox and is asserted against in the same `pytest` invocation — no scheduler, no daemon, no `__main__` entry point, no network, no broker SDK, no real operator, no real account / venue / endpoint identifier, no real secret. ADR-005's runtime concerns (the operator-availability daemon, the external sink, the scheduler) all remain out of scope and out of the diff.
3. The `KillSwitchProtocol` and `ReconciliationProtocol` surfaces, by contrast, are nearer to the always-forbidden categories in `MASTER_STATUS.md` §6 (broker execution, order management, account fetch, fills) and require more review attention to keep a first fixture from drifting across the line. Starting at `HeartbeatProtocol` lets the first fixture exercise the test-fixture pattern itself (proof-bundle shape, allowlist update mechanics, Mode A / Mode B sequencing under ADR-008 / ADR-009) under the smallest blast radius.

**This recommendation does not bind Kevin's choice.** Kevin may name `KillSwitchProtocol` or `ReconciliationProtocol` as the first protocol in writing at the time of P3-03 implementation authorization. Whichever protocol Kevin names, the scope properties below apply unchanged.

**Scope properties (apply to whichever single protocol Kevin names):**

- **One protocol, not all three.** The first P3-03 implementation PR targets exactly one of `KillSwitchProtocol`, `ReconciliationProtocol`, or `HeartbeatProtocol`. The other two remain untouched and require their own separate written authorizations to be added later.
- **Test-fixture only.** The implementation is a test-fixture artifact under `tests/**` and, where strictly necessary, a sandboxed in-memory fake helper under a new `src/gmc_rebuild/<protocol>/` subdirectory specifically scoped to test-fixture support (named by Kevin in writing at the time of P3-03 implementation authorization; see "Expected Future Files / Directories" below for the planning-level naming convention). The fixture is exercised under `pytest`. No `__main__` entry point. No daemon. No scheduler. No background worker. No network. No broker SDK. No real account / venue / endpoint identifier. No real secret. No real or historical market-data feed. No order object. No order placement. No external sink. No persistence outside the test sandbox.
- **One control surface at a time.** Even within the chosen protocol, the test-fixture exercise is narrow enough that a single review can verify it does not cross into trading strategy, broker execution, live or paper trading wired to a real broker, real market-data ingestion, real secrets, order placement, scheduler / daemon, persistence outside the test sandbox, or runtime-daemon territory — i.e. the always-forbidden categories in `MASTER_STATUS.md` §6 and the stop conditions in `plan/phase3_entry_plan.md` §8.
- **In-memory fake helper, not real runtime.** Any sandboxed helper added under `src/gmc_rebuild/<protocol>/` is an in-memory fake whose only consumers are the test fixtures themselves. It is not imported by any `__main__` entry point, by any daemon, by any scheduler, by any broker integration, or by any real-runtime path. It is not exported by `src/gmc_rebuild/__init__.py` as part of any runtime API.
- **No relaxation of the P2-05 boundary.** `src/gmc_rebuild/risk/` continues to contain only the abstract `typing.Protocol` definitions and supporting frozen dataclasses / enums for the three control surfaces. The fake helper added (if any) lives under a separate `src/gmc_rebuild/<protocol>/` subdirectory specifically authorized in the P3-03 implementation PR, not inside `src/gmc_rebuild/risk/`.
- **Reversibility.** The P3-03 implementation PR is reversible by a single `git revert` on the merge commit. The proof bundle per `plan/phase3_entry_plan.md` §6 demonstrates this explicitly.
- **Allowlist update in the same PR.** Any new directory introduced by the future P3-03 implementation must be added to the `MASTER_STATUS.md` §8 step 4a `allowed_p2_infra` allowlist in the same PR that introduces the directory, per the standing rule in `MASTER_STATUS.md` §8 step 4b. This planning PR does **not** extend the allowlist; the planning-level reference below is non-operative and pending authorization.

## Expected Review Risks for the Future P3-03 Implementation PR (Planning Level Only)

The future P3-03 implementation PR is the **first concrete behaviour behind a P2-05 control surface**. Even though the implementation is restricted to a test-fixture artifact and a sandboxed in-memory fake helper, the review carries elevated risk relative to a documentation-only PR. The following risks are identified at the planning level so the Mode A review of the future P3-03 implementation PR has a concrete checklist to work against. None of these risks is mitigated by this planning PR; each must be re-verified against the actual diff of the P3-03 implementation PR.

1. **Scope creep across the always-forbidden line.** The greatest risk is that the test-fixture artifact, however small, accidentally introduces a name, an import, a dependency, or a code path that crosses into one of the always-forbidden categories (strategy, signals, scanners, models, portfolio, backtests, broker, execution, live, paper, daemons, market_data, orders, secrets). Mode A review must explicitly verify §8 step 4 (top-level) and §8 step 4c (recursive tokenized) scans return clean on the implementation branch.
2. **`__main__` introduction.** A fake helper is not a runtime, but a careless `if __name__ == "__main__":` block (even one that only prints) creates a runtime entry point and crosses the boundary. Mode A review must verify no `__main__` entry point exists anywhere in the implementation diff.
3. **Scheduler / daemon / background-worker introduction.** A fake helper that uses `threading.Thread`, `asyncio.create_task`, `time.sleep` inside a loop, or any APScheduler-style construct crosses the boundary even if no broker SDK is imported. Mode A review must verify no scheduler / daemon / background-worker construct exists anywhere in the implementation diff.
4. **Filesystem persistence outside the test sandbox.** A fake helper that writes to disk outside `tmp_path` / `tmpdir` (e.g. writes to `~/.gmc_rebuild/`, opens a SQLite file, opens a log sink) crosses the boundary. Mode A review must verify no on-disk persistence exists outside the test sandbox.
5. **Network call.** A fake helper that imports `socket`, `urllib`, `requests`, `httpx`, or any broker / market-data SDK crosses the boundary even if the call is never executed. Mode A review must verify no network-capable import exists in the implementation diff.
6. **Real secret / credential / account identifier / venue identifier / endpoint leakage.** Even a fake helper might accidentally embed a string that looks like a real broker account, a real venue identifier, or a real endpoint. Mode A review must verify the `detect-secrets` baseline is unchanged in a way that masks new findings, that `.secrets.baseline` is regenerated correctly if at all, and that no string in the diff resembles a real credential / account / venue / endpoint.
7. **Concrete implementation inside `src/gmc_rebuild/risk/`.** The P2-05 boundary states that `src/gmc_rebuild/risk/` contains types and abstract `typing.Protocol` definitions only. A fake helper accidentally added inside `src/gmc_rebuild/risk/` (rather than under a new, allowlist-updated `src/gmc_rebuild/<protocol>/`) violates that boundary. Mode A review must verify the fake helper lives outside `src/gmc_rebuild/risk/`.
8. **Allowlist drift.** Adding the new `src/gmc_rebuild/<protocol>/` directory **without** updating the `allowed_p2_infra` allowlist (and the matching comment block / `case "$path"` mapping) in the same PR is a STOP condition. Adding it to the allowlist **without** Kevin's separate written authorization is also a STOP condition. Mode A review must verify both the allowlist update and the durable authorization artifact under `governance/authorizations/` are present in the implementation PR.
9. **Quality-gate relaxation.** Any change to `.pre-commit-config.yaml`, `pyproject.toml`, `.gitignore`, `.secrets.baseline`, the `MASTER_STATUS.md` §8 step 4 always-forbidden scan, the §8 step 4a allowlist scan, or the §8 step 4c recursive forbidden-token scan that weakens a check is a STOP condition. Mode A review must verify no such relaxation exists in the implementation diff.
10. **Mode A / Mode B sequencing failures.** The Mode A critique must be against the P3-03 implementation PR itself (not against this planning PR, not against the Phase 3 entry decision, not against any earlier artifact). The Mode B monitoring packet for the P3-03 implementation PR's active workday must be authored, committed in a separate sibling monitoring PR, and merged to `main` **before** the P3-03 implementation PR merges per ADR-008 §D5 (or ADR-009 D5 once in force). A merge-order inversion (implementation PR merged before its Mode B sibling) is a STOP condition.
