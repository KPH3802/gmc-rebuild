### 3.1 Candidate Objective

The candidate P6-05 task introduces a **deterministic in-memory simulated portfolio state** that applies accepted P6-03 / P6-04 outputs to a position book under **explicit, deterministic, fixture-only fill assumptions**. The portfolio is a pure value object (or a pure transformation producing one): there is **no real market data, no broker confirmation, no real account, no persistence, and no runtime activation**. The fill assumptions are caller-supplied fixtures — for example, a deterministic full-fill-at-limit-price assumption, or a caller-supplied fill price and fill quantity — and are part of the explicit, closed inputs; the candidate task does not infer fills from any external source. The candidate task is a structural application of already-rendered upstream decisions to a value-typed book; it is **not** strategy logic, **not** order routing, and **not** execution.

### 3.3 Candidate State Model Decision — To Be Made Before Implementation

The implementation authorization must choose, in writing, **before any implementation work begins**, between two planning-level candidate state models. This planning packet does **not** choose:

- **Candidate model (1) — Frozen / value-typed replaceable snapshot.** The portfolio is a frozen, slotted value object (a closed map of symbol → position, plus whatever closed summary fields the implementation authorization names). "Applying" an accepted intent is a pure transformation that **returns a new snapshot**; the prior snapshot is never mutated. State evolution is expressed by replacing one immutable snapshot with the next. This mirrors the frozen/slotted dataclass discipline already established by the merged P5-02 `SimulatedOrderIntent`, P6-01 `SignalIntent`, P6-02 `EligibilityDecision`, and P6-03 `PositionDecision`.
- **Candidate model (2) — Event-style application model.** The portfolio is reconstructed (or folded) from an ordered sequence of accepted-intent application events. The "current" state is a deterministic function of the event sequence. Each application appends an event; the snapshot is derived by replaying / folding.

The choice between candidate model (1) and candidate model (2) materially affects the idempotence contract in §3.3.1 and the equality / identity semantics in the test matrix (§3.8). Per Kevin's verbatim authorization, **the implementation authorization names the chosen model**; this planning packet records both as planning-level candidates without pre-committing.

#### 3.3.1 Candidate Idempotence Decision — To Be Made Before Implementation

The implementation authorization must name, in writing, the idempotence contract for repeated application of the **same** accepted intent:

- Under candidate model (1) (frozen replaceable snapshot), the natural candidate is **idempotent application**: applying the same accepted intent (identified by its deterministic P6-04 intent identifier) to the same snapshot a second time yields a snapshot equal to the first application's result — the duplicate is recognized and absorbed, not double-counted.
- Under candidate model (2) (event-style), application may instead be **explicitly non-idempotent**: a second application appends a second event and the folded state reflects two applications, unless the implementation authorization specifies a deduplication rule keyed on the deterministic intent identifier.

Per Kevin's verbatim authorization, **the later implementation authorization must name the chosen idempotence model** (idempotent, or explicitly non-idempotent with a stated deduplication rule or stated absence of one). This planning packet records both behaviours as planning-level candidates and pre-commits to neither. The deterministic P6-04 `derive_simulated_order_intent_id` identifier is the natural deduplication key if the implementation authorization chooses an idempotent or dedup-on-identity contract.

### 3.6 Deterministic Application Contract (Planning Level Only)

The candidate application function has the following contract at the planning level. All elements are planning-level candidates; the implementation authorization names the exact contract.

- **Determinism.** Identical inputs (same prior snapshot / event sequence, same accepted decision, same intent, same fill-assumption fixture) return the byte-for-byte identical output snapshot every call. No global state, no module-level mutable, no cache, no monotonic counter, no `random`, no `time`, no environment lookup, no clock read, no UUID-from-entropy.
- **No mutation of inputs.** The supplied prior snapshot, decision, intent, and fixtures are not modified in place. All upstream dataclasses are already frozen / slotted; this contract preserves that and additionally requires the new portfolio value type to be frozen / slotted.
- **Explicit, fixture-only fills.** The fill price and fill quantity are taken from the caller-supplied fixture, never inferred from any external source. There is no market-data lookup, no quote, no order book, no broker confirmation.
- **No hidden state.** The application function holds no module-level mutable state, no class-level mutable state, no instance state. It is a closed transformation from input values to an output value.
- **No side effects.** No log emission inside the function itself (or, if a structured `audit_event` record is later desired, that is a P6-06-shaped concern explicitly out of scope here). No filesystem, no network, no broker SDK, no order placement, no market-data lookup, no time-source read, no environment-variable read, no `time.sleep`, no `os.environ` / `os.getenv`.
- **Closed import surface.** The internal module imports only `__future__`, `dataclasses` (stdlib), `enum` (stdlib), `typing` / `collections.abc` (stdlib) as needed, and the already-merged in-repo modules required to consume the typed inputs (`gmc_rebuild.decision`, `gmc_rebuild.simulation`, and optionally `gmc_rebuild.signal_intake` / `gmc_rebuild.eligibility` / `gmc_rebuild.runtime`). No import of `gmc_rebuild.config`, `os`, `socket`, `requests`, `urllib`, `http`, `threading`, `asyncio`, `subprocess`, `sqlite3`, `time`, `pickle`, `shelve`, `ssl`, `smtplib`, `ftplib`, `uuid`, `random`, or any third-party dependency.

- Maintain any **real position book**, real account balance, real holdings, or real P&L. The simulated portfolio is a pure value object computed from explicit fixtures; it tracks nothing outside its own inputs and connects to nothing.
- Place any broker call. No broker SDK import, no broker session, no broker reconciliation against a real broker, no broker credential, no broker reference of any kind.
- Place any real or paper order. No paper-broker connector, no paper-account identifier, no paper-API surface, no live-broker connector, no live-account identifier, no live-API surface, no execution loop.
- Introduce any real venue identifier, real account identifier, real broker identifier, real routing instruction, or real endpoint URL. The carried-forward `symbol` fields remain free-form `str` values carried by value; the portfolio does not validate them against any external universe.
- Mutate any input. The supplied prior snapshot, `PositionDecision`, `SimulatedOrderIntent`, fill-assumption fixture, and any carried-forward values are not modified in place.
- Hold any hidden state. No module-level mutable, no class-level mutable, no cache, no counter, no log buffer, no in-memory queue beyond the value objects the caller explicitly passes and receives.
- Connect to any market. No market-data SDK, no market-data ingestion, no historical data feed, no quote feed, no order book, no FIX / REST / WebSocket client. All fills are caller-supplied fixtures.
- Activate any runtime. No `__main__` entry point, no daemon, no scheduler, no background thread, no long-running service, no `time.sleep`, no `asyncio.sleep`, no cron job, no launchd plist. No re-export from `src/gmc_rebuild/__init__.py`; the public surface is reachable only via `from gmc_rebuild.<name> import ...`.
- Persist anything. No SQLite, no DuckDB, no on-disk database, no on-disk portfolio snapshot, no filesystem snapshot, no on-disk audit trail, no log file, no file artifact written from any runtime path. No `open(` builtin, no `pickle`, no `shelve`.
- Load any external configuration. No import of `gmc_rebuild.config`, no `.env` reading, no YAML / TOML / JSON loading from disk. Every input is supplied by the caller.
- Modify any merged surface. P6-05 introduces only the new sibling subpackage; it does not modify the merged P5-01 / P5-02 / P6-04 `SimulationBoundary` / `SimulatedOrderIntent` / `SimulatedOrderTimeInForce` / `derive_simulated_order_intent_id` surface, the merged P6-01 `SignalIntent` / `SignalSide`, the merged P6-02 `EligibilityOutcome` / `EligibilityReason` / `EligibilityConfig` / `EligibilityDecision` / `check_eligibility`, the merged P6-03 `PositionDecisionOutcome` / `PositionDecisionReason` / `PositionDecision` / `compose_position_decision`, the merged P4-06 `SafetyVerdict.clear` precondition, the merged P4-08 `BLOCKER_*` set, or any merged P2 / P3 / P4 / P5 / P6 file outside the new directory.
- Implement any concrete protocol. No concrete `HeartbeatProtocol` / `KillSwitchProtocol` / `ReconciliationProtocol` implementation outside the merged P3-03 / P3-04 / P3-05 in-memory fakes.
- Implement strategy logic. The simulated portfolio is a structural application of already-rendered upstream decisions to a value-typed book; it does not generate signals, does not scan markets, does not score candidates, does not rank alternatives, does not pick which intent to apply, does not size positions strategically, and does not pick a venue or order type. Position arithmetic under explicit fixtures is bookkeeping, not strategy.
