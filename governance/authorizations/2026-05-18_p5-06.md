  - Asserts the following invariant categories: (1) clear-path symmetry — on a real-pipeline-clear `SafetyVerdict` both `propose(intent=..., verdict=verdict)` and `propose_order(order_intent=..., verdict=verdict)` return the supplied argument **by identity**, so the two methods agree on accept; (2) lane-mismatch symmetry — a lane-mismatched `SimulatedIntent` (resp. `SimulatedOrderIntent`) raises `SimulationBoundaryError` from `propose` (resp. `propose_order`) with the boundary lane and the intent lane both reflected in the message, with no dependence on `verdict.clear`; (3) verdict-type symmetry — passing a non-`SafetyVerdict` argument as `verdict` raises `SimulationBoundaryError` from each method with the wrong-type name in the message; (4) intent-type symmetry — passing a non-`SimulatedIntent` to `propose` (resp. non-`SimulatedOrderIntent` to `propose_order`) raises `SimulationBoundaryError` with the wrong-type name in the message; (5) blocker-tuple propagation symmetry — for each of the five `BLOCKER_*` codes driven from the corresponding P3 fake state, both `propose` and `propose_order` raise `SimulationBoundaryError` whose message contains the same `verdict.blockers` tuple representation, and the blocker code itself appears in both error messages; (6) multi-blocker propagation symmetry — when heartbeat STALE, kill switch TRIPPED, and reconciliation FAILED are simultaneously true, both `propose` and `propose_order` raise errors whose messages name every blocker code; (7) non-mutation symmetry on the clear path — neither boundary method mutates the supplied intent, the `SafetyVerdict`, the boundary's lane, or any P3 fake state across repeated calls; (8) non-mutation symmetry on the blocked path — when a verdict is blocked, neither boundary method mutates the supplied intent, the `SafetyVerdict`, the boundary's lane, or any P3 fake state across repeated calls; (9) re-evaluation symmetry — after staging a CLEAN reconciliation outcome that follows a FAILED outcome, the next `evaluate()` produces a clear verdict and both boundary methods again accept by identity; (10) an inertness self-check using `ast.walk` to confirm the new module's actual imports are disjoint from `{os, socket, requests, urllib, http, threading, asyncio, subprocess, sqlite3, pickle, shelve, ssl, smtplib, ftplib}` and that every imported module starts with one of the authorized prefixes.
