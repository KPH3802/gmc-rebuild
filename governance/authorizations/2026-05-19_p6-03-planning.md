### 3.2 Typed Input Boundary

The decision composer accepts the following typed inputs, all by value, all already-constructed by the caller:

- A `SignalIntent` instance from the merged P6-01 typed signal-intake boundary (`from gmc_rebuild.signal_intake import SignalIntent`). The closed five-field shape (`intent_id`, `symbol`, `side`, `quantity`, `rationale`) is consumed unchanged; no expansion of the P6-01 dataclass is contemplated by this packet.
- An `EligibilityDecision` instance from the merged P6-02 eligibility-check pure functions (`from gmc_rebuild.eligibility import EligibilityDecision`). The closed two-field shape (`outcome: EligibilityOutcome`, `reasons: tuple[EligibilityReason, ...]`) is consumed unchanged; no expansion of the P6-02 dataclass is contemplated by this packet.
- Already-constructed implementations of the merged P3-03 / P3-04 / P3-05 in-memory Protocol fakes — `InMemoryHeartbeat` (`from gmc_rebuild.heartbeat import InMemoryHeartbeat`), `InMemoryKillSwitch` (`from gmc_rebuild.kill_switch import InMemoryKillSwitch`), and `InMemoryReconciliation` (`from gmc_rebuild.reconciliation import InMemoryReconciliation`) — supplied by the caller (typically a test fixture) via dependency injection. The composer does not instantiate, configure, or runtime-activate these fakes; it consumes them through their abstract `typing.Protocol` boundaries declared in `src/gmc_rebuild/risk/`.
- Optionally, an already-constructed `RuntimeShell` instance from the merged P4-06 safety boundary (`from gmc_rebuild.runtime import RuntimeShell`). The candidate design either (a) accepts a `RuntimeShell` directly and calls its read-only `evaluate()` method once to obtain a `SafetyVerdict`, or (b) accepts the three Protocol fakes plus an already-built `SafetyVerdict` produced upstream. Kevin's future P6-03 implementation authorization picks one of (a) / (b) explicitly; this planning packet records both as planning-level candidates without pre-committing.

No other input type is contemplated. No `EligibilityConfig` is consumed (eligibility is the upstream P6-02 layer's responsibility; the decision composer reads only the already-rendered `EligibilityDecision`). No raw configuration slice, no environment variable, no filesystem, no broker session, no market-data snapshot, no order book, no portfolio snapshot, no time source — every input is supplied by the caller as a value of a closed, frozen, already-validated type.

### 3.3 Typed Output Boundary

The decision composer returns a single frozen, slotted dataclass — provisionally `PositionDecision` — with exactly the closed-shape fields needed to express a "would-trade" or "would-skip" outcome:

- `outcome: PositionDecisionOutcome` — a new closed two-member `StrEnum` declaring exactly `WOULD_TRADE` and `WOULD_SKIP`. No third outcome member. Naming is a planning-level candidate; the future P6-03 implementation authorization may substitute e.g. `TRADE` / `SKIP` or `ALLOW` / `BLOCK`, but the closed two-member shape is preserved.
- `reasons: tuple[PositionDecisionReason, ...]` — a tuple of closed-set reason codes (see §3.4 below) explaining the decision. Empty on the `WOULD_TRADE` path; non-empty and sorted in the canonical declaration order of `PositionDecisionReason` on the `WOULD_SKIP` path. Enforces the biconditional `outcome == WOULD_TRADE iff reasons == ()` at construction, mirroring the merged P6-02 `EligibilityDecision` precedent.
- `intent_id: str` — the `SignalIntent.intent_id` carried forward by value for trace correlation. The composer does not generate or transform this identifier; it copies it.
- `verdict: SafetyVerdict` — the upstream P4-06 safety verdict consumed by the composer, carried forward by value so callers (and tests) can inspect what safety state drove the outcome. This field is the read-only handle into the merged P4-06 safety surface; the composer does not re-implement, modify, or shadow `SafetyVerdict`.

The exact field set above is a planning-level candidate. Kevin's future P6-03 implementation authorization may rename `verdict`, may collapse `intent_id` and `verdict` into a single composed "decision context" record, or may add an `eligibility: EligibilityDecision` field carrying the upstream P6-02 result by value. The closed-shape principle and the `WOULD_TRADE iff reasons == ()` biconditional are not negotiable at the planning level; everything else is.

### 3.4 Reason Taxonomy for Trade and Skip Outcomes

The candidate `PositionDecisionReason` closed `StrEnum` enumerates every reason the composer can append to the `reasons` tuple of an outcome. Each reason is structural — every reason is decidable from the typed inputs alone — and exhaustively covers the disjoint failure modes propagated from the upstream layers. The candidate (planning-level only) member set is:

- `ELIGIBILITY_INELIGIBLE` — propagated when the supplied `EligibilityDecision.outcome` is `EligibilityOutcome.INELIGIBLE`. The accompanying `EligibilityDecision.reasons` tuple is carried forward by value on the `PositionDecision.eligibility` field (if §3.3 chooses to add that field) or referenced via the input, but is **not** re-encoded into individual `PositionDecisionReason` members; the upstream P6-02 reason taxonomy is preserved unchanged.
- `SAFETY_HEARTBEAT_STALE` — propagated when the upstream `SafetyVerdict` reports any required component as not `FRESH` (per ADR-005). One reason member covers all heartbeat staleness states; individual component identities are preserved through the carried-forward `SafetyVerdict` for operator-view inspection.
- `SAFETY_KILL_SWITCH_NOT_ARMED` — propagated when the upstream `SafetyVerdict` reports the kill-switch as not `ARMED` (per ADR-002).
- `SAFETY_RECONCILIATION_NOT_CLEAN` — propagated when the upstream `SafetyVerdict` reports reconciliation as not `CLEAN` (per ADR-003). The `WARNING` advisory-grade blocker preserved by the merged P4-08 safety-policy-hardening tests is reported through this reason; the upstream `SafetyVerdict` carries the distinguishing `UNAVAILABLE` / `FAILED` / `WARNING` state for operator-view inspection.
- `SAFETY_VERDICT_NOT_CLEAR` — propagated when the upstream `SafetyVerdict.clear` is `False` for any reason not enumerated above (defense-in-depth against future `BLOCKER_*` constant additions that the composer might not name individually). This member exists so that a future `BLOCKER_*` addition does not silently produce a `WOULD_TRADE` decision when safety is in fact blocked.

The closed set above maps one-to-one onto the merged P4-08 `BLOCKER_*` constants exposed by the P4-06 `RuntimeShell` / `SafetyVerdict` surface plus the P6-02 `EligibilityOutcome.INELIGIBLE` propagation. Adding a sixth reason member is a P6-03 implementation-authorization decision, not a planning decision. Removing any of the five named reason members would weaken the composed-pipeline coverage and is not contemplated by this packet.

The biconditional that `outcome == WOULD_TRADE` exactly when (`EligibilityDecision.outcome == ELIGIBLE` **and** `SafetyVerdict.clear` is `True`) is the load-bearing composed-pipeline invariant. Every reason member above corresponds to a specific way that biconditional can be violated; the composer surfaces every applicable failing reason in canonical declaration order on the `WOULD_SKIP` path.

### 3.5 Deterministic Pure-Function Contract

The composer is exposed as a single pure function — provisionally `compose_position_decision(intent, eligibility, verdict) -> PositionDecision` (signature is a planning-level candidate; the future P6-03 implementation authorization names the exact signature, e.g. it may accept a `RuntimeShell` instead of a pre-built `SafetyVerdict`, or it may accept the three Protocol fakes directly). The contract is:

- **Determinism.** Identical inputs return structurally-equal `PositionDecision` instances every call. No global state, no module-level mutable, no cache, no monotonic counter, no `random`, no `time`, no environment lookup.
- **Identity preservation of inputs.** The supplied `SignalIntent`, `EligibilityDecision`, `RuntimeShell` (if direct), and `SafetyVerdict` are returned unchanged by identity — i.e. the composer does not mutate them and does not return modified copies of them.
- **No mutation of inputs.** All upstream dataclasses are already frozen / slotted; the composer asserts this fact at construction time of the output and does not attempt any in-place modification. Repeated calls with the same upstream instances do not perturb any observable state in the P6-01 signal-intake surface, the P6-02 eligibility surface, the P3-03 / P3-04 / P3-05 in-memory fakes, the P4-06 / P4-07 / P4-08 safety surface, or the P5-01..P5-07 simulation surface.
- **No hidden state.** The composer holds no module-level mutable state, no class-level mutable state, no instance state outside the returned frozen `PositionDecision`. The function is a closed transformation from input values to an output value.
- **No side effects.** No log emission inside the composer itself (the P6-02 audit trail will, in a later packet, be threaded through the merged `gmc_rebuild.logging.audit_event` helper from a separate caller-side concern — but the planning-level decision is that the composer itself does not log, does not emit, does not record). No filesystem, no network, no broker SDK, no order placement, no market-data lookup, no time source read, no environment-variable read, no `time.sleep`, no `os.environ` / `os.getenv`.
- **Closed import surface.** The internal `_compose` module imports only `__future__`, `dataclasses` (stdlib), `enum` (stdlib), and the already-merged in-repo modules `gmc_rebuild.signal_intake` (for `SignalIntent` / `SignalSide`), `gmc_rebuild.eligibility` (for `EligibilityDecision` / `EligibilityOutcome` / `EligibilityReason`), `gmc_rebuild.risk` (for the abstract `typing.Protocol` definitions if dependency-injection is via Protocol references), and `gmc_rebuild.runtime` (for `RuntimeShell` / `SafetyVerdict`). No import of `gmc_rebuild.config`, `gmc_rebuild.simulation`, `os`, `socket`, `requests`, `urllib`, `http`, `threading`, `asyncio`, `subprocess`, `sqlite3`, `time`, `pickle`, `shelve`, `ssl`, `smtplib`, `ftplib`, or any third-party dependency.

### 3.6 Safety and Non-Goals

The candidate P6-03 packet explicitly **does not**:

- Place any broker call. No broker SDK import, no broker session, no broker reconciliation, no broker credential, no broker reference of any kind. The decision composer is upstream of any order; it does not connect to anything.
- Execute any live or paper trade. No paper-broker connector, no paper-account identifier, no paper-API surface, no paper-trading execution loop. No live-broker connector, no live-account identifier, no live-API surface, no live-trading execution loop.
- Mutate any input. The supplied `SignalIntent`, `EligibilityDecision`, `RuntimeShell`, `SafetyVerdict`, `InMemoryHeartbeat`, `InMemoryKillSwitch`, and `InMemoryReconciliation` instances are not modified in place. All upstream dataclasses are already frozen / slotted; the composer preserves that.
- Hold any hidden state. No module-level mutable, no class-level mutable, no cache, no counter, no log buffer, no decision queue, no audit ledger.
- Connect to any market. No market-data SDK, no market-data ingestion, no historical data feed, no quote feed, no order book, no FIX / REST / WebSocket client. The `symbol` field of `SignalIntent` is a free-form `str` carried forward by value; the composer does not validate it against any external universe.
- Activate any runtime. No `__main__` entry point, no daemon, no scheduler, no background thread, no long-running service, no `time.sleep`, no `asyncio.sleep`, no cron job, no launchd plist. No re-export from `src/gmc_rebuild/__init__.py`; the composer's public surface is reachable only via `from gmc_rebuild.decision import ...`.
- Persist anything. No SQLite, no DuckDB, no on-disk database, no on-disk decision snapshot, no on-disk audit trail, no log file, no file artifact written from any runtime path. No `open(` builtin, no `pickle`, no `shelve`.
- Load any external configuration. No import of `gmc_rebuild.config`, no `.env` reading, no YAML / TOML / JSON loading from disk. Every input is supplied by the caller.
- Modify any merged surface. No modification of the merged P6-01 `signal_intake/` source files, no modification of the merged P6-02 `eligibility/` source files, no modification of the merged P5-01..P5-07 `simulation/` surface, no modification of the merged P4-06 / P4-07 / P4-08 `runtime/` surface, no modification of the merged P3-03 / P3-04 / P3-05 in-memory fakes, no modification of any merged P2 / P3 / P4 / P5 / P6 file.
- Implement any concrete protocol. No concrete `HeartbeatProtocol` / `KillSwitchProtocol` / `ReconciliationProtocol` implementation outside the merged P3-03 / P3-04 / P3-05 in-memory fakes; the composer consumes the existing fakes by dependency injection.
- Implement strategy logic. The composer is a structural composition of already-rendered upstream decisions; it does not generate signals, does not scan markets, does not score candidates, does not rank alternatives, does not pick which intent to compose, does not produce a position size, and does not produce a venue / order-type / time-in-force choice.

### 3.7 Proposed Test Matrix and Invariants

The candidate test module at `tests/decision/test_decision.py` (and any additional composed-edge-case test files mirroring the P4-02 / P4-03 / P4-04 / P4-05 precedent) is expected, at the planning level only, to cover the following invariant categories. Each category is a planning-level candidate; Kevin's future P6-03 implementation authorization may add, remove, or replace any category.

1. **Would-trade happy path.** Given an `ELIGIBLE` `EligibilityDecision` and a `clear` `SafetyVerdict`, the composer returns `PositionDecision(outcome=WOULD_TRADE, reasons=(), …)` with the supplied `intent_id` and the supplied `verdict` carried forward by value.
2. **Would-skip on ineligibility.** Given an `INELIGIBLE` `EligibilityDecision` and a `clear` `SafetyVerdict`, the composer returns `outcome=WOULD_SKIP` and `reasons` includes `ELIGIBILITY_INELIGIBLE`. The five upstream `EligibilityReason` codes from `EligibilityDecision.reasons` are preserved through the carried-forward eligibility reference; the composer does not re-encode them into its own taxonomy.
3. **Would-skip on each `BLOCKER_*`.** For each of the five merged P4-08 `BLOCKER_*` constants (heartbeat stale, kill-switch not armed, reconciliation `WARNING`, reconciliation not `CLEAN` in any non-`WARNING` failure mode, plus the catch-all "not clear" case), assert that an `ELIGIBLE` eligibility plus the matching non-clear `SafetyVerdict` produces `outcome=WOULD_SKIP` and a `reasons` tuple containing exactly the matching `PositionDecisionReason` member.
4. **Would-skip on composed failure.** Given an `INELIGIBLE` `EligibilityDecision` **and** a non-`clear` `SafetyVerdict`, assert that `outcome=WOULD_SKIP` and `reasons` includes both `ELIGIBILITY_INELIGIBLE` and the matching `SAFETY_*` member(s), in the canonical declaration order of `PositionDecisionReason`.
5. **Biconditional invariant.** `outcome == WOULD_TRADE iff reasons == ()` is asserted at the dataclass-construction level (rejecting `WOULD_TRADE` with non-empty `reasons` and rejecting `WOULD_SKIP` with empty `reasons` at `__post_init__`).
6. **Closed-set enumeration invariants.** `set(PositionDecisionOutcome) == {WOULD_TRADE, WOULD_SKIP}`; `set(PositionDecisionReason) == {ELIGIBILITY_INELIGIBLE, SAFETY_HEARTBEAT_STALE, SAFETY_KILL_SWITCH_NOT_ARMED, SAFETY_RECONCILIATION_NOT_CLEAN, SAFETY_VERDICT_NOT_CLEAR}` (or whatever the authorized closed set is); both enumerations are `StrEnum` subclasses; values equal names.
7. **Frozen / slotted / closed-shape invariants.** `PositionDecision.__slots__` exists; assignment to a field after construction raises `FrozenInstanceError`; the instance has no `__dict__`; the declared field tuple equals exactly the authorized set in order; the declared types match the documented shape.
8. **Determinism / repeat-call invariant.** Calling `compose_position_decision` with the same inputs twice returns structurally-equal `PositionDecision` instances; no observable state in the upstream P6-01 / P6-02 / P3 / P4 / P5 surfaces is perturbed by repeated calls.
9. **Non-mutation of inputs.** Before and after `compose_position_decision`, the supplied `SignalIntent`, `EligibilityDecision`, and `SafetyVerdict` snapshots are unchanged (asserted by deep equality on snapshot copies).
10. **Identity-return of inputs (where applicable).** The composer returns the supplied `SafetyVerdict` and the supplied `SignalIntent` by identity-preservation into the output (`PositionDecision.verdict is verdict_input`; `PositionDecision.intent_id == intent.intent_id` and is the same `str` object where the language guarantees identity).
11. **Type validation on inputs.** Passing a non-`SignalIntent` for `intent`, a non-`EligibilityDecision` for `eligibility`, or a non-`SafetyVerdict` / non-`RuntimeShell` for the safety argument raises `TypeError` with a message naming the offending type.
12. **Equality and hashability of `PositionDecision`.** Structurally-equal instances compare `==` and have equal hashes; outcome-differing or reasons-differing instances do not compare equal.
