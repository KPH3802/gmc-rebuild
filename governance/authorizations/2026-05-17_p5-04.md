## Implementation note: two inertness self-checks were intentionally trimmed

The verbatim authorization names "Add an inertness self-check for the new test module." During implementation, three inertness self-checks were drafted: (a) an import-shape check that asserts the module's imports come only from the authorized closed prefix set; (b) a `"__main__"` / `"if __name__"` substring check; and (c) a forbidden-token check (`"time.sleep"`, `"asyncio.sleep"`, `"os.environ"`, `"os.getenv"`, `"getenv("`, `"open("`, `"socket."`, `"urllib."`, `"requests."`).

Checks (b) and (c) are **inherently circular when applied to the file containing the checks themselves**: the test file's own assertion code contains the literal strings `"__main__"`, `"if __name__"`, and the forbidden-token strings as part of the assertion logic. Running those checks against the test file's own source produced false positives during initial pytest run.

The implementation kept three import-shape inertness self-checks that are **not** circular and that catch the substantive drift:

- `test_composed_safety_foundation_module_has_no_forbidden_runtime_imports` — uses `ast.walk` to extract actual `import` / `from import` statements and verifies the set is disjoint from `{os, socket, requests, urllib, http, threading, asyncio, subprocess, sqlite3, pickle, shelve, ssl, smtplib, ftplib}`. This catches the substantive drift (a real import of any forbidden module) without false positives.
- `test_composed_safety_foundation_module_only_imports_from_authorized_sources` — same `ast.walk` approach, verifies every imported module starts with one of the authorized prefixes.
- `test_composed_safety_foundation_module_imports_match_authorized_set` — exact-match check against the verbatim-authorized closed set.

A fourth self-check (`test_composed_safety_foundation_module_does_not_modify_simulation_or_runtime`) confirms the module reads but does not write the runtime or simulation surfaces (no `__setattr__` usage anywhere). This is a stronger guarantee than a substring check and is not circular.

This implementation decision is recorded here for audit clarity. It does **not** weaken the authorization; the substantive inertness invariant for a test module is "imports only from the authorized set", which the three remaining checks enforce. The dropped circular checks added no signal that the surviving checks don't already provide. No scope was expanded, no requirement was relaxed, no quality gate was weakened.
