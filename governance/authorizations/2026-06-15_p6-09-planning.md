### 3.1 Candidate Objective

The candidate P6-09 task introduces a **deterministic, in-memory, read-only position reconciliation comparison** for the dry-run engine. It is a **pure projection of two already-rendered value-typed snapshots** — the merged P6-05 `SimulatedPortfolio` snapshot and a caller-supplied `ExpectedPositions` value object representing the per-symbol net positions the operator expects to see — into a frozen, typed reconciliation result that downstream surfaces could optionally consume in a separately-authorized future packet (e.g. a future implementation could extend the P6-06 `DailyReport` and / or the P6-07 `DryRunOperatorView` to carry P6-09 result fields; that extension is **not** authorized by this packet).

The comparison answers two questions per symbol: (a) is the symbol present in both snapshots, and (b) do the signed net quantities match. The aggregate result is a closed `DryRunReconciliationOutcome` enum (`MATCH` when every symbol matches and no extras; `MISMATCH` otherwise) plus per-symbol detail (`matches`, `quantity_mismatches`, `only_in_simulated`, `only_in_expected`). The result also echoes a caller-supplied `ReconciliationStatus` value (the merged `gmc_rebuild.risk.ReconciliationStatus` enum) so the planning-packet-required ADR-003 `UNAVAILABLE` vs `FAILED` distinction is preserved end-to-end (the caller chooses whether to pass `CLEAN`, `WARNING`, `UNAVAILABLE`, or `FAILED` based on whether expected data was available; this packet does not authorize any code path that itself selects between those values).

It computes a structured result and stops. There is **no I/O, no external sink, no file artifact, no scheduler, no `__main__`, no persistence, no network, no live data, and no runtime activation.** The result is a value object with optional rendered fields; what the caller does with it (echo into a future daily-report field, echo into a future operator-view section, log it, ignore it) is the caller's choice and outside this capability. The comparison is **strictly read-only**: it constructs nothing that feeds back into the engine, mutates none of its inputs, and changes no decision, intent, portfolio, report, or operator view.

### 3.3 Candidate Value-Object Shape — To Be Made Before Implementation

The implementation authorization must choose, in writing, **before any implementation work begins**, the exact value-object shape. This planning packet records the candidate shape without pre-committing:

- **Result value object (frozen / slotted).** A new frozen, slotted dataclass (planning-level candidate name `DryRunReconciliationResult`) with the closed field set:
  - `outcome: DryRunReconciliationOutcome` — closed enum (`MATCH` / `MISMATCH`); enforced biconditional `MATCH iff quantity_mismatches == () and only_in_simulated == () and only_in_expected == ()`.
  - `matches: tuple[tuple[str, int], ...]` — canonical (sorted by symbol, unique, non-zero quantity) tuple of `(symbol, net_quantity)` pairs that appear in both snapshots with equal signed net quantities.
  - `quantity_mismatches: tuple[ReconciliationQuantityMismatch, ...]` — canonical (sorted by symbol, unique) tuple of frozen, slotted records per symbol where both snapshots carry an entry but the quantities differ; each record carries `symbol`, `simulated_quantity`, and `expected_quantity`.

  - `only_in_expected: tuple[tuple[str, int], ...]` — canonical tuple of `(symbol, net_quantity)` pairs present in the `ExpectedPositions` input but absent from the `SimulatedPortfolio` snapshot.

Construction is value-only — the value object accepts a pre-validated tuple of `(symbol, signed_net_quantity)` pairs. The implementation authorization may optionally name an `ExpectedPositions.from_simulated_portfolio(snapshot) -> ExpectedPositions` convenience constructor (planning-level candidate) that synthesizes an `ExpectedPositions` from a `SimulatedPortfolio` (for tests and self-comparison sanity checks); this convenience does **not** open any real-account path.

### 3.7 Deterministic Construction Contract (Planning Level Only)

All elements are planning-level candidates; the implementation authorization names the exact contract.
