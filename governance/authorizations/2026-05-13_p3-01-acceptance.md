ADR-009's substantive Decision sections (D1 through D7), Alternatives Considered, Consequences, Implementation Notes, Follow-up Actions, and Related ADRs are **not** rewritten by this acceptance. The Mode A re-review against the PR #32 revision returned a satisfactory result (B1–B5 resolved, no scope drift introduced, no new blocking findings), so no further substantive revision is bundled with acceptance.
- Two narrow, related clarifications that the Mode A re-review flagged as non-blocking improvements (NB-NEW-1 and NB-NEW-2 in the re-review summary). Each clarification is a short prose tweak inside an existing ADR-009 paragraph; neither changes the rule, neither expands scope, neither creates a new control surface:
  - **NB-NEW-1 — Weekend-runtime edge under the D3 weekday fallback.** D3's fallback clause ("scheduled active on every UTC weekday … until a committed schedule lands on `main`") is read by the re-review to silently exempt UTC weekends for a runtime process that lacks a committed schedule but actually runs over a weekend. The clarification states that a weekend run still satisfies condition 2 (it ran for some portion of the UTC day), and therefore the day is a runtime day and a packet is required, independent of the weekday fallback's "scheduled active" reading. The fallback is a conservative floor that adds days, not a cap that subtracts them.
  - **NB-NEW-2 — D5 catch-up naming, reason (iii), hybrid case.** D5's reason (iii) currently asserts that "filename collision is structurally avoided because the missed day is by definition a day on which no `YYYY-MM-DD.md` packet was committed", but the immediately-following sentence acknowledges a hybrid case where a governance packet for the day exists yet the runtime portion was missed. The clarification labels reason (iii) explicitly as "pure-miss case", retains the hybrid case acknowledgement immediately after, and reconciles the two so a reader does not read (iii) as denying the hybrid case it is then asked to handle.
- Updating `docs/decisions/ADR-008_monitoring_cadence_and_ai_monitor_role.md` "Follow-up Actions" first entry to **check off** the §D6 runtime monitoring cadence follow-up, with a pointer to ADR-009 in Accepted status.
